# 전국 보호구역 모니터링 및 지도 대시보드 범위

작성일: 2026-07-23

## 방향

서울시 보호구역 변경 감지가 정상 동작하는 것을 기준으로, 수집 범위를 전국으로 확장하고 변경 이력을 대시보드와 지도에서 확인할 수 있게 만든다. 1차 가시화 범위는 보호구역 Polygon과 시설 Point로 제한하고, 표준노드링크 또는 NGIS 링크 매칭 결과는 이후 같은 구조 위에 확장한다.

## 1. 전국 보호구역 수집 확장

### 1-1. 행정 코드 관리

- 서울 고정 코드 목록 대신 공식 법정동/행정구역 코드 원천 CSV에서 현재 유효한 시군구 5자리 코드를 생성한다.
- 행정구역 개편으로 코드가 변경될 수 있으므로, 코드 파일은 수동 고정값이 아니라 공식 원천을 기준으로 재생성 가능해야 한다.
- 품질 점검에서 설정된 시군구 코드가 현재 DB 수집 결과에 모두 존재하는지 확인한다.
- 특정 시군구 응답이 0건이면 대량 삭제로 처리하지 않고 실행 실패로 중단한다.

### 1-2. 전국 단위 수집 및 등록

- 전국 코드 파일을 `SGG_CODES_FILE`로 지정해 전체 수집을 수행한다.
- 최초 전국 실행은 기존 current 상태를 덮어쓰기 전에 API 응답 건수, 시군구별 커버리지, Polygon/Point 유효성, 중복 관리번호를 확인한다.
- 최초 등록 이후부터는 동일한 변경 감지 로직으로 신규/변경/삭제 이벤트를 누적한다.

## 2. 대시보드 및 지도 표시

현재 구현된 1차 산출물:

- `safety-zone-monitor export-dashboard` 명령으로 운영 DB의 JSON/GeoJSON 파일을 생성한다.
- `dashboard/index.html`에서 현재 Polygon/Point와 변경 Polygon/Point 레이어를 표시한다.
- 최근 실행과 최근 변경 이벤트를 검색·필터링할 수 있다.
- 지도 레이어 선택은 운영 상태 기준인 `현재`, `신규`, `변경`, `삭제(검토)`로 분류한다.

### 2-1. 최초 신규 추가 지역 등록

- 서울 외 지역이 처음 들어오는 전국 통판 적재는 신규 이벤트가 아니라 기준선 `현재`로 등록한다.
- 전국 기준선 이후 발생한 신규만 `신규` 이벤트로 저장한다.
- 대시보드에서는 실행 ID, 실행 시간, 수집 범위, 신규 Polygon/Point 수를 확인할 수 있어야 한다.

### 2-2. 변경 이력 확인

- 실행 단위 이력: 성공/실패, 수집 건수, Polygon/Point 변경 요약.
- 이벤트 단위 이력: 신규, 위치/도형 변경, 속성 변경, 위치/도형+속성 변경, 삭제, 응답 누락 검토.
- 검색/필터: 시설명, 관리번호, 시군구 코드, 변경 유형, 실행 ID.
- 지도 표시: Polygon은 면, Point는 마커로 표시하고 변경 유형별 색상과 팝업 상세를 제공한다.

## 3. 링크 매칭 결과 확장 구조

- 1차 지도 레이어는 보호구역 Polygon과 시설 Point만 사용한다.
- 추후 `analysis.zone_link_match_candidate_v2`, `analysis.zone_link_match_excluded_v2`, NGIS 링크 결과를 별도 레이어로 추가한다.
- 보호구역 그룹 ID를 기준으로 Polygon/Point/표준링크/NGIS 링크를 같은 상세 화면에서 조회할 수 있게 한다.

## 4. 포인트 삭제와 누락 판정

기존에는 현재 Point가 신규 응답에서 사라지면 모두 `MISSING`으로 분류했다. 하지만 같은 실행에서 연결된 보호구역 Polygon 그룹이 삭제된 경우는 시설 폐쇄 또는 보호구역 해제로 인한 정상 삭제에 가깝다.

따라서 포인트 분류 기준은 다음과 같이 둔다.

| 유형 | 기준 | 운영 해석 |
|---|---|---|
| `DELETED` | Point가 사라졌고 같은 `zone_group_id`의 Polygon 삭제 이벤트가 있음 | 시설 폐쇄/보호구역 해제 가능성이 높은 삭제 |
| `MISSING` | Point가 사라졌지만 같은 그룹의 Polygon 삭제 근거가 없음 | API 응답 누락, 수집 이상, 원천 데이터 정합성 검토 대상 |

예를 들어 키즈위즈유치원처럼 Polygon 삭제와 Point 사라짐이 같은 그룹에서 동시에 발생하면 `Point DELETED`로 보는 것이 적절하다. 반대로 Polygon 삭제 근거 없이 Point만 사라진 상원유치원, 미광유치원 같은 케이스는 `MISSING`으로 남겨 검토 대상으로 표시한다.
