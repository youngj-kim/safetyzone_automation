# 전국 통판 기준선 적재 정책

작성일: 2026-07-23

## 문제

서울 모니터링이 먼저 운영된 상태에서 전국 보호구역 통판을 처음 수집하면, 기존 current 테이블에 없던 서울 외 지역이 모두 `NEW`로 감지될 수 있다. 하지만 운영자가 보고 싶은 것은 "전국 최초 적재로 인해 새로 DB에 들어온 데이터"가 아니라 "기준선 이후 실제로 변경된 보호구역"이다.

따라서 전국 확장 시에는 전국 데이터 자체를 변경 이벤트로 보지 않고, 기준선 current로 먼저 올리는 절차가 필요하다.

## 기준선 제안

- 전 버전 기준일: 2026-07-07
- 전국 기준선 상태: 2026-07-22까지 존재하던 전국 보호구역은 `현재`로 간주
- 변경 감시 기준일: 2026-07-23부터 발생한 신규/변경/삭제만 변경 이력으로 표시

현재 대시보드에서 2026-07-07 실행을 전 버전으로 볼 수 있으므로, 서울 구간은 2026-07-07 상태를 과거 기준으로 두고 2026-07-23 감지분을 변경 이력으로 유지한다.

대시보드의 `신규`는 우리 DB에 처음 들어온 시점이 아니라 기준선 이후 API 본 서버에 새로 나타난 대상을 의미한다. 따라서 2026-07-07 기준 API에 이미 존재하던 시설은 설령 우리 DB에 뒤늦게 적재되더라도 `신규`가 아니라 `현재`로 표시한다.

## 권장 절차

1. 전국 시군구 코드 파일을 생성한다.
2. 전국 보호구역 데이터를 별도 기준선 모드로 수집한다.
3. 기준선 모드는 snapshot/current는 채우되 `zone_change_event`, `zone_facility_point_change_event`, 알림은 만들지 않는다.
4. 이 기준선 적재분은 대시보드에서 `현재` 레이어로만 노출한다.
5. 기준선 이후 정기 실행부터 `NEW`, `ATTRIBUTE_CHANGED`, `GEOMETRY_CHANGED`, `POINT_CHANGED`, `DELETED`, `MISSING` 이벤트를 생성한다.
6. 2026-07-23 서울 변경 감지분처럼 실제 감지된 이벤트만 `신규`, `변경`, `삭제(검토)`로 표시한다.

## 대시보드 분류

| 대시보드 분류 | 포함 데이터 | 의미 |
|---|---|---|
| `현재` | `analysis.zone_current`, `analysis.zone_facility_point_current` | 기준선 또는 최신 실행 기준의 현재 유효 데이터 |
| `신규` | `NEW` 이벤트 | 기준선 이후 새로 생긴 보호구역/시설 Point |
| `변경` | `ATTRIBUTE_CHANGED`, `GEOMETRY_CHANGED`, `GEOMETRY_ATTRIBUTE_CHANGED`, `POINT_CHANGED`, `POINT_ATTRIBUTE_CHANGED` | 기준선 이후 속성, 도형, 위치가 바뀐 대상 |
| `삭제(검토)` | `DELETED`, `MISSING` 이벤트 | 삭제 또는 수집/원천 정합성 검토 대상 |

## `삭제(검토)` 하위 판정

`MISSING` Point는 운영 해석이 케이스별로 다르므로, 대시보드에서 현재 그룹 상태를 교차 확인해 검토 사유를 함께 표시한다.

| 검토 사유 | 기준 | 예시 해석 |
|---|---|---|
| Polygon도 함께 삭제됨 | 같은 관리번호의 Polygon `DELETED` 이벤트가 있음 | 시설/보호구역 삭제 가능성이 높음 |
| 같은 그룹 Polygon 유지 | Point는 사라졌지만 같은 `zone_group_id`의 현재 Polygon 또는 Point가 남아 있음 | 대표구역 병합, 부속시설 Point 정리, 원천 속성 정리 가능성 |
| 현재 그룹 없음 | Point가 사라졌고 현재 그룹도 없음 | 원천 삭제 또는 응답 누락 확인 필요 |

2026-07-23 서울 변경분 기준으로는 키즈위즈유치원은 Polygon과 Point가 함께 사라져 삭제 가능성이 높고, 상원유치원은 청계초등학교 그룹이 현재 유지되므로 Point 누락/정리 검토가 필요하다. 미광유치원은 태릉초등학교 그룹에 묶여 있고 태릉초등학교 변경 이벤트가 함께 있으므로 대표구역 병합 또는 부속 Point 삭제로 보는 것이 자연스럽다.

## 반복 감지 운영 규칙

같은 관리번호가 여러 실행에 걸쳐 계속 변할 수 있으므로, 이벤트는 덮어쓰지 않고 누적한다. 대시보드는 Polygon/Point와 `source_manage_no`를 기준으로 타임라인을 묶어 최근 상태를 표시한다.

| 상태 | 기준 | 운영 의미 |
|---|---|---|
| `삭제 검토 1회` | 최신 이벤트가 `MISSING`이고 연속 누락 1회 | 원천 응답 누락, Point 정리, 일시 오류 가능성을 검토 |
| `반복 누락, 삭제 의심` | 같은 대상의 최신 이벤트가 `MISSING`이고 연속 누락 2회 이상 | 삭제 후보로 격상하되 시군구 수집 성공 여부와 그룹 상태 확인 |
| `삭제 확인` | 최신 이벤트가 `DELETED` | Polygon 삭제 또는 삭제로 확정 분류된 Point |
| `누락 후 재등장` | 과거 `MISSING` 이후 최신 이벤트가 `NEW` 또는 변경 이벤트 | 원천 복구, 재등록, 관리번호 재사용 가능성 확인 |
| `변경 추적` | 최신 이벤트가 속성/도형/위치 변경 | API 최종수정일과 시스템 감지일을 함께 확인 |

이 규칙은 `analysis.zone_change_event`, `analysis.zone_facility_point_change_event`에 남은 감지 이력을 기반으로 하며, 변경이 없는 실행은 이벤트를 만들지 않는다. 따라서 "현재 유지"는 current 테이블 기준이고, "반복 누락"은 실제로 누락 이벤트가 반복 생성된 경우에만 판단한다.

## 구현 시 주의점

- 전국 기준선 적재는 일반 `run`과 분리된 명령으로 두는 것이 안전하다.
- 기준선 적재는 기존 변경 이벤트를 지우지 않아야 한다.
- 서울에서 이미 감지된 2026-07-23 변경 이벤트는 유지해야 한다.
- 기준선 이후 삭제 검토는 같은 시군구가 정상 응답했다는 수집 커버리지 검증을 통과한 경우에만 생성해야 한다.
- API가 과거 날짜별 스냅샷을 제공하지 않는다면, "어제까지 기준"은 운영상 승인한 최초 전국 수집 시점의 current 기준선으로 정의해야 한다.
